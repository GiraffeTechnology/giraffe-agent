# CLAUDE.md — Giraffe Agent M-side MVP as OpenClaw-compatible Supplier Skill

# Task: Build a real M-side MVP that connects to the existing B-side MVP and enables supplier-side execution through OpenClaw / WeChat / WhatsApp / IM channels

# Product target: M-side MVP after B-side MVP. The final result must support a runnable B+M procurement execution loop.

# Date: 2026-05-23

---

## 0. Read This First

You are working inside the existing Giraffe Agent repository (`giraffe-agent`, Python 3.11+, FastAPI + Pydantic v2, uv package manager).

The B-side MVP has already been specified as an OpenClaw-compatible AI Buyer skill. This M-side MVP must extend that architecture and create the supplier-side product layer.

The goal is to build a **real M-side MVP**, not a static demo, not a supplier portal mockup, and not a generic chatbot.

The M-side MVP must allow SME suppliers / factories / workshops to receive buyer enquiries and respond through IM-native channels such as WeChat, WhatsApp, OpenClaw, email-style text input, or the fallback web workspace.

The M-side MVP must support this minimum supplier loop:

1. Receive a structured supplier inquiry generated by the B-side AI Buyer.
2. Create or resume a supplier-side workspace linked to the B-side enquiry workspace.
3. Identify the supplier and supplier session through channel user ID, invitation token, supplier ID, or manual selection.
4. Ask the supplier for missing capacity / schedule / material / quote / QC / logistics fields.
5. Convert natural-language supplier replies into a structured `SupplierResponsePacket`.
6. Return the structured supplier response back to the B-side workspace.
7. Allow the B-side AI Buyer to run delivery feasibility simulation using real M-side responses.
8. If a supplier is selected, create a minimal post-confirmation execution workspace.
9. Let the supplier acknowledge the order and submit production node updates, QC image notes, exception reports, logistics handover, and completion confirmation through IM.
10. Log all M-side events into the Industrial Execution Graph v0.1 seed.

The M-side MVP must be designed so that when it is connected with the B-side MVP, the system can run a real minimum B-to-M procurement execution workflow:

```text
Buyer IM input
  → B-side structured requirement
  → supplier inquiry package
  → supplier dispatch
  → M-side supplier response
  → B-side delivery feasibility simulation
  → buyer selects delivery path
  → M-side order acknowledgement
  → production / QC / logistics updates
  → execution events logged
```

Do NOT rewrite the repo from scratch. Extend the existing code.

## 2. MVP definition

### 2.1 What this M-side MVP must prove

The MVP must prove that an SME supplier can use Giraffe Agent as an AI Merchandiser / Supplier Response Agent through an IM-native workflow.

The supplier should be able to complete this minimum product loop:

1. Receive a buyer inquiry through OpenClaw / WeChat / WhatsApp / Web.
2. Open or resume a supplier-side inquiry workspace.
3. Understand what the buyer is asking for.
4. Reply in natural language or structured text.
5. Confirm capacity, schedule, material availability, quote, MOQ, lead time, QC ability, packaging, logistics readiness, and risks.
6. Submit a structured supplier response packet to the B-side workspace.
7. If selected, acknowledge the order.
8. Submit production milestone updates.
9. Submit QC / image / video notes as text references or attachment metadata.
10. Report exceptions.
11. Confirm logistics handover and completion.
12. Store all supplier-side activity as Industrial Execution Graph v0.1 events.

The M-side MVP is real if and only if a supplier can use IM to move from inquiry receipt to structured response submission, and then to basic order execution updates after selection.

### 2.2 What this M-side MVP is not

- It is not a full supplier ERP.
- It is not a full MES integration.
- It is not a supplier marketplace.
- It is not a payment, escrow, wallet, or settlement system.
- It is not a production-grade factory operating system.
- It is not a complex supplier onboarding portal.
- It is not a full computer vision QC system.
- It is not a static investor demo.
- It is not a generic chatbot.

### 2.3 Required product language

Use these terms consistently in code comments, docs, and UI:
