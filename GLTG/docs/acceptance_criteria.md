# GLTG Acceptance Criteria

This document lists the 20 acceptance criteria for the GLTG engine. Each item is marked as **Implemented** with a brief note on how it is satisfied.

---

| # | Criterion | Status | Implementation Note |
|---|---|---|---|
| 1 | Given a fully populated `ApparelOrderInput` with 5 participants, the engine must return a `DeliveryFeasibilityPacket` with `status = FEASIBLE` or `LIMITED_OPTIONS`, a non-null `commitable_date`, and at least one `DeliveryPathOption`. | **Implemented** | Verified by `run_10000_shirts_acceptance.py`. The engine's evaluate pipeline builds the graph, enumerates options, prunes, ranks, and assembles the packet. |
| 2 | Given zero participants, the engine must return `status = NO_FEASIBLE_OPTION` and `options = []`. | **Implemented** | `FeasibilityPacketBuilder.build()` sets `NO_FEASIBLE_OPTION` when `n == 0` options. Verified by `run_zero_one_two_supplier_cases.py`. |
| 3 | Given exactly one garment factory participant, the engine must return `status = LIMITED_OPTIONS` with exactly one `DeliveryPathOption` and `LIMITED_COMPETITION` in `risk_flags`. | **Implemented** | The participant cap rule (`max_options = min(3, count)`) limits to 1 option. `FeasibilityPacketBuilder` adds `LIMITED_COMPETITION` flag. Verified by acceptance script. |
| 4 | Given exactly two garment factory participants, the engine must return `status = LIMITED_OPTIONS` with exactly two `DeliveryPathOption` objects and `LIMITED_COMPARISON` in `risk_flags`. | **Implemented** | Same cap mechanism, `FeasibilityPacketBuilder` adds `LIMITED_COMPARISON`. Verified by acceptance script. |
| 5 | Duration estimates must incorporate evidence from supplier memory records when available, producing `memory_adjusted_days` on the `DurationEstimate`. | **Implemented** | `DurationEstimator.estimate()` queries `supplier_memory` for matching `participant_id` and `node_type`, computes `memory_adjusted_days`, and includes a `HISTORICAL_MEMORY` evidence item. |
| 6 | Duration estimates must incorporate evidence from `SupplierResponse` confirmations, producing `supplier_claim_days` on the `DurationEstimate`. | **Implemented** | `DurationEstimator.estimate()` uses the matching `SupplierResponse.confirmed_days` as `supplier_claim_days` with a `SUPPLIER_CONFIRMATION` evidence item at weight 0.85. |
| 7 | The engine must identify and return a non-empty `critical_path` list of node IDs for any order with at least one participant. | **Implemented** | `CriticalPathFinder.find()` performs a backward-pass computation on the resolved graph. `packet.critical_path` is populated from the best option or the raw graph. |
| 8 | The engine must identify `bottleneck_nodes` -- nodes with the lowest float (slack) on the critical path. | **Implemented** | `CriticalPathFinder.find_bottlenecks()` identifies nodes where total float equals zero or is minimal. Stored in `graph.metadata["bottleneck_nodes"]` and propagated to the packet. |
| 9 | The 6-tier evidence hierarchy must be used for duration blending, with `ACTUAL_PROGRESS` at weight 1.0 and `AI_ESTIMATE` at weight 0.25. | **Implemented** | `EvidenceWeighter.EVIDENCE_WEIGHTS` defines the full tier table. `EvidenceWeighter.blend()` computes the weighted average. |
| 10 | The engine must return probabilistic duration estimates with p50, p80, and p90 percentile bands on every `DurationEstimate`. | **Implemented** | All `DurationEstimate` objects include `p50_days`, `p80_days`, `p90_days`. Baselines from `apparel/baselines.py`; refined by estimator. |
| 11 | The SEWING node duration must scale linearly with order quantity using the baseline of 7 days per 1,000 pieces. | **Implemented** | `_sewing_baseline(quantity)` in `baselines.py` applies the formula. The estimator uses capacity-based calculation as a refinement when `capacity_per_day` is provided. |
| 12 | The engine must support working-day calendars with configurable working days per week, holiday exclusions, and timezone. | **Implemented** | `CalendarConfig` model accepted by `ApparelOrderInput`. `DependencyResolver` and `GraphBuilder` use `CalendarConfig` for date arithmetic via `graph/calendars.py`. |
| 13 | The path enumerator must generate batch-split and alternative-route variants (air freight, stock fabric) in addition to base options. | **Implemented** | `BatchSplitAnalyzer.generate_splits()` and `AlternativeRouteGenerator.generate()` produce variants. Both are called inside `LeadTimeGraphEngine.evaluate()`. |
| 14 | Options must be ranked using an 8-factor scoring formula and labelled FASTEST, MOST_RELIABLE, or BEST_COMMERCIAL_BALANCE. | **Implemented** | `OptionRanker.rank()` applies the 8-factor formula. `_assign_labels()` assigns semantic labels. Verified in `option_ranker.py`. |
| 15 | The reforecast API must accept a list of `ProgressEvent` objects, apply them to the existing packet's graph, and return an updated packet with revised `commitable_date`. | **Implemented** | `LeadTimeGraphEngine.reforecast()` delegates to `ReforecastEngine.reforecast()`, which applies events via `EventApplier` and re-resolves the graph. |
| 16 | When reforecast shows slippage greater than 7 days, a `TIGHT_DEADLINE` risk flag must be added to the updated packet. | **Implemented** | `ReforecastEngine.reforecast()` checks `delta_days > 7` and appends a `TIGHT_DEADLINE` `RiskFlag` with `severity="HIGH"`. |
| 17 | The engine must surface expedite levers (air freight, overtime, stock fabric, etc.) when schedule slippage is detected during reforecast. | **Implemented** | `ExpediteOptionGenerator.generate()` is called when `delta_days > 0`. Six levers are available, sorted by `days_saved`. |
| 18 | All models must be Pydantic v2 `BaseModel` subclasses, fully serializable to JSON via `model_dump(mode="json")`. | **Implemented** | All models in `gltg/models/` inherit from `pydantic.BaseModel`. `serializers.py` uses `model_dump(mode="json")`. Round-trip tested via `deserialize_packet(serialize_packet(packet))`. |
| 19 | The engine must return identical `status` and option count across 5 consecutive evaluations of the same order on the same day. | **Implemented** | Verified by `verify_gltg_5x.py`. Engine is stateless; `date.today()` is stable within a day. |
| 20 | The `gltg evaluate` and `gltg reforecast` CLI commands must be functional and produce parseable output. | **Implemented** | `gltg/integrations/cli.py` implements both subcommands using `argparse`. Full JSON, `--summary` human-readable, and `-o FILE` output modes are supported. |
